/orders/gross-profit-margin-from-order-voucher
Biên lợi nhuận gộp & ROI từng loại voucher
Mô tả: "Mỗi đồng bỏ ra cho voucher, thu về bao nhiêu GMV?"
▼
Mục đích nghiệp vụ

Endpoint này giải quyết bài toán đo hiệu quả chi tiêu marketing qua voucher. Seller trên TikTok Shop có thể chạy 3 loại voucher khác nhau: voucher từ nền tảng TikTok (platform trợ giá), voucher do seller tự bỏ tiền, và voucher miễn phí vận chuyển từ nền tảng. Mỗi loại có nguồn chi và đối tượng chịu phí khác nhau.

Biên lợi nhuận gộp (Gross Profit Margin) phản ánh lợi nhuận trước chi phí vận hành và marketing — không tính chi phí cố định. Đây là chỉ số nhanh để biết sản phẩm có đang bán lỗ ở cấp độ đơn hàng hay không.

⚠
Lưu ý quan trọng:

Một đơn hàng có thể dùng đồng thời nhiều loại voucher. Code dùng 3 if độc lập (không phải else-if) nên GMV của đơn đó sẽ được cộng vào tất cả bucket voucher mà nó thuộc về. Đây là overlap có chủ đích để ROI của mỗi nguồn chi phản ánh đúng thực tế.

Công thức tính Gross Profit Margin
Tính cho mỗi đơn hàng
costCommission = totalAmount × (commissionRate / 100) / 100

grossProfit += totalAmount
             − skuSalePrice   ← // giá vốn hàng bán (cost of goods)
             − shippingFee    ← // phí vận chuyển seller chịu
             − costCommission ← // hoa hồng trả cho creator
Công thức tính ROI từng loại voucher
3 bucket độc lập
// Bucket 1: Platform voucher (TikTok trợ giá cho buyer)
if voucherDeductionPlatform > 0:
  totalGmvPlatform += order.totalAmount
  totalVoucherPlatform += voucherDeductionPlatform

// Bucket 2: Seller voucher (seller tự bỏ tiền trợ giá)
if voucherDeductionSeller > 0:
  totalGmvSeller += order.totalAmount
  totalVoucherSeller += voucherDeductionSeller

// Bucket 3: Free shipping voucher từ platform
if shippingFeeDeductionPlatformVoucher > 0:
  totalGmvShipping += order.totalAmount
  totalVoucherShipping += shippingFeeDeductionPlatformVoucher

ROI = totalGmv / totalVoucherSpend
// ROI = 25 nghĩa là 1đ chi → 25đ GMV
Tại sao innerJoin thay vì leftJoin?

innerJoinAndSelect với điều kiện OR trên 3 trường voucher hoạt động như bộ lọc ngầm: chỉ đơn hàng có ít nhất một loại voucher > 0 mới xuất hiện trong kết quả. Dùng leftJoin sẽ đưa vào cả đơn không có voucher nào, làm loãng tập dữ liệu tính GPM.

TypeScript3 if độc lập — intentional overlap
// 3 điều kiện ĐỘC LẬP — một đơn có thể cộng vào cả 3 bucket
if (Number(order.voucher_deduction_platform) > 0) {
  totalGmvPlatform    += Number(order.total_amount);
  totalVoucherPlatform += Number(order.voucher_deduction_platform);
}
if (Number(order.voucher_deduction_seller) > 0) {
  totalGmvSeller    += Number(order.total_amount);
  totalVoucherSeller += Number(order.voucher_deduction_seller);
}
if (Number(order.shipping_fee_deduction_platform_voucher) > 0) {
  totalGmvShipping    += Number(order.total_amount);
  totalVoucherShipping += Number(order.shipping_fee_deduction_platform_voucher);
}

  .createQueryBuilder('o')
  .leftJoinAndSelect(

Mục đích nghiệp vụ

Endpoint này cho phép Sales/BD nhìn vào con số sau khi đã trừ chi phí gửi mẫu, phản ánh sát hơn với P&L thực tế của chiến dịch KOC. Ví dụ: GMV = 200 triệu, chi phí sample = 15 triệu → Doanh thu thực = 185 triệu.

Công thức
actualRevenue = SUM(order_payments.total_amount) − totalCostSampleOrders

// GMV chỉ tính đơn:
// - source = 'AFFILIATE' (đơn từ link affiliate creator)
// - cancel_time IS NULL (chưa bị seller huỷ)
// - cancel_reason IS NULL (chưa có lý do hoàn trả)
// - creator_username IS NOT NULL (có creator)

Luồng thực thi: 2 queries tuần tự — (1) SQL tính tổng GMV affiliate với điều kiện lọc, (2) gọi getSampleProductCosts(userId, dto) với cùng bộ tham số để lấy totalCostSampleOrders. Trừ (2) khỏi (1) trả về kết quả.

SQLTổng GMV affiliate — loại cancel, loại hoàn
SELECT SUM(p.total_amount) AS "actualRevenue"
FROM   orders o
LEFT JOIN order_payments p ON o.order_id = p.order_id
WHERE  o.create_time       BETWEEN $1 AND $2
  AND  o.cancel_time        IS NULL   -- chưa bị seller huỷ
  AND  o.cancel_reason      IS NULL   -- chưa có lý do hoàn trả
  AND  o.creator_username   IS NOT NULL
  AND  o.source             = 'AFFILIATE'
  AND  o.created_by         = $3

Mục đích nghiệp vụ

AOV (Average Order Value) là chỉ số nhanh để đánh giá sức mua trung bình của khách hàng đến từ kênh affiliate/KOC. AOV thấp hơn kỳ vọng có thể là dấu hiệu creator đang thu hút sai tệp khách hàng, hoặc sản phẩm bundle cần được cải thiện.

Chỉ tính đơn source = 'AFFILIATE' có creator username — loại đơn shop thông thường ra để đo riêng hiệu quả kênh KOC.

Công thức
AOV = ROUND( SUM(payment) / NULLIF(COUNT(DISTINCT order_id), 0), 2 )

// NULLIF tránh chia cho 0 khi không có đơn nào
SQLLATERAL — đảm bảo 1 payment per order
SELECT
  ROUND(
    SUM(p.payment) / NULLIF(COUNT(DISTINCT o.order_id), 0),
    2
  ) AS "averageOrderValue"
FROM orders o
LEFT JOIN LATERAL (
  -- correlated subquery: chạy riêng cho mỗi row orders
  SELECT payment FROM order_payments
  WHERE  order_id = o.order_id
  LIMIT  1  -- lấy đúng 1, tránh fan-out
) p ON true
WHERE o.is_sample_order = false
  AND o.source          = 'AFFILIATE'
  AND o.creator_username IS NOT NULL
  AND o.buyer_user_id   IS NOT NULL
  AND o.status          NOT IN ('CANCELLED')

Mục đích nghiệp vụ

Trong TikTok Shop, thời điểm đăng video ảnh hưởng trực tiếp đến tỉ lệ chuyển đổi. Creator/seller muốn biết khung giờ nào trong ngày hệ thống ghi nhận nhiều đơn nhất — để lên lịch flash deal, đẩy thông báo hay nhắc creator đăng video đúng lúc.

Endpoint trả về 4 thông tin: today.goldenFrame (khung giờ vàng hôm nay), total.goldenFrame (tổng hợp toàn lịch sử), today.hourlyStats (array 24 phần tử số đơn theo giờ), và bestDay (ngày có tổng đơn cao nhất trong kỳ).

today.goldenFrame
Khung giờ vàng hôm nay — format "H–H+windowSize". Dùng để ra quyết định realtime.
total.goldenFrame
Khung giờ vàng tổng hợp toàn lịch sử trong kỳ. Dùng cho chiến lược dài hạn.
today.hourlyStats
Array 24 phần tử — số đơn theo từng giờ hôm nay. Dùng để vẽ biểu đồ phân phối.
bestDay
Ngày có tổng đơn cao nhất trong date range. Dùng để phân tích sự kiện/trend theo ngày.

Thuật toán Sliding Window: với windowSize = 1 (cố định trong code hiện tại), tìm giờ h sao cho tổng đơn trong khoảng [h, h + windowSize) là lớn nhất. Về mặt kỹ thuật, đây là bài toán "maximum subarray sum" trên mảng 24 phần tử — được giải bằng SQL Window Function thay vì vòng lặp để tận dụng index.

Sliding Window
win_sum(h) = SUM(orders) trong [h, h + windowSize)

goldenHour = argmax(win_sum)
goldenFrame = "goldenHour:00 – (goldenHour + windowSize):00"

// generate_series(0, 23) đảm bảo đủ 24 giờ
// kể cả giờ không có đơn nào (COUNT = 0)
ℹ
Tại sao dùng generate_series?

Nếu giờ 3 AM không có đơn nào, query GROUP BY sẽ không trả về row cho giờ đó. Sliding window sẽ bị sai vì mảng thiếu phần tử. generate_series(0, 23) tạo đủ 24 rows trước rồi LEFT JOIN với data thực — giờ không có đơn sẽ có COUNT = 0 thay vì bị bỏ qua.

PostgreSQL CTEWindow Function — ROWS BETWEEN CURRENT ROW AND N FOLLOWING
WITH hours AS (
  -- đảm bảo đủ 24 rows, kể cả giờ không có đơn
  SELECT generate_series(0, 23) AS hour
),
agg AS (
  SELECT
    h.hour,
    COUNT(b.hour) FILTER (WHERE b.is_today) AS today_total,
    COUNT(b.hour)                               AS total_all
  FROM hours h LEFT JOIN base_all b ON b.hour = h.hour
  GROUP BY h.hour
),
win_today AS (
  SELECT
    hour,
    SUM(today_total) OVER (
      ORDER BY hour
      -- sliding window: cộng windowSize giờ liên tiếp
      ROWS BETWEEN CURRENT ROW AND $2 - 1 FOLLOWING
    ) AS win_sum
  FROM agg
)
SELECT hour, win_sum FROM win_today
ORDER BY win_sum DESC, hour ASC LIMIT 1
-- ORDER BY hour ASC khi win_sum bằng nhau: ưu tiên giờ sớm hơn

Mục đích nghiệp vụ

Đây là "bảng hiệu quả" của từng creator trong danh sách hợp tác. Seller cần biết: creator nào đã nhận sample, đã đăng video chưa, và video đó tạo ra bao nhiêu đơn thực tế (không phải sample). Thông tin này dùng để ra quyết định tái hợp tác — creator nào ROI tốt thì gửi thêm sample, creator nào nhận sample mà không tạo đơn thì cắt giảm.

Điểm phức tạp: cần map từ productCreatorContents.productId (sản phẩm trong content) sang orderLineItems.productId (sản phẩm trong đơn hàng) để xác định video nào đã được đăng cho sản phẩm nào.

Luồng 2 queries tuần tự, sau đó merge in-memory. Toàn bộ sort, filter, paginate chạy sau khi đã load hết data vào RAM — không phân trang ở DB.

2

3

4

// Bước 1: Thu thập tất cả productId có content liên kết
const postedProductIds = new Set<string>();
for (const occ of order.orderCreatorContents ?? []) {
  for (const pcc of occ?.creatorContent?.productCreatorContents ?? []) {
    if (pcc?.productId) postedProductIds.add(pcc.productId.toString());
  }
}

// Bước 2: Kiểm tra từng sản phẩm trong đơn hàng
for (const item of order.orderLineItems) {
  if (!item.productId) continue;
  const isPosted = postedProductIds.has(item.productId.toString());

  // Chỉ đếm unposted khi đơn DELIVERED hoặc COMPLETED
  if (!isPosted && ['DELIVERED', 'COMPLETED'].includes(order.status)) {
    stats.totalUnpostedVideo++;
  }
}

Mục đích nghiệp vụ

Endpoint này cung cấp bức tranh tổng thể funnel KOC theo đơn hàng: bao nhiêu đơn sample đã giao → bao nhiêu creator đã đăng video → trong số đó bao nhiêu video tạo ra đơn hàng thực có thanh toán (hasPaidOrder = true) vs. chỉ tạo đơn chưa thanh toán.

Đơn vị đếm là đơn hàng (order), không phải video hay creator. Một creator có thể có nhiều đơn sample; mỗi đơn được phân loại độc lập.

ℹ
Bucket ưu tiên hasPaid:

Nếu một đơn vừa có content hasPaidOrder = true vừa có content hasPaidOrder = false, đơn đó chỉ được tính vào bucket hasPaid. Logic: if (hasPaid) count++; if (!hasPaid && hasNoPaid) countNoPaid++ — không phải else-if đơn giản.

const contents = order.orderCreatorContents
  ?.map(ooc => ooc.creatorContent)

Tại sao LEFT JOIN IS NULL thay vì NOT IN?

NOT IN (subquery) phải re-evaluate subquery cho mỗi row — với bảng lớn sẽ rất chậm và không dùng được index. LEFT JOIN + IS NULL chỉ scan một lần, index-friendly hơn, và cũng xử lý đúng edge case khi subquery trả về NULL.

WITH old_customers AS (
  SELECT DISTINCT buyer_user_id
  FROM  orders
  WHERE create_time  < $1         -- toàn bộ lịch sử trước startTime
    AND  source       = 'AFFILIATE'
    AND  creator_username IS NOT NULL
    AND  buyer_user_id    IS NOT NULL
    AND  status        != 'CANCELLED'
    AND  created_by    = $3
),
period_customers AS (
  SELECT
    o.buyer_user_id,
    (oc.buyer_user_id IS NULL) AS is_new_customer
  FROM  orders o
  LEFT JOIN old_customers oc ON o.buyer_user_id = oc.buyer_user_id
  WHERE o.create_time BETWEEN $1 AND $2
    AND  o.source     = 'AFFILIATE'
    AND  o.status    != 'CANCELLED'
    AND  o.created_by = $3
)
SELECT
  COUNT(DISTINCT buyer_user_id) AS "totalUniqueCustomers",
  COUNT(DISTINCT CASE WHEN is_new_customer THEN buyer_user_id END) AS "newUniqueCustomers",
  ROUND(
    COUNT(DISTINCT CASE WHEN is_new_customer THEN buyer_user_id END)::numeric
    / NULLIF(COUNT(DISTINCT buyer_user_id), 0) * 100, 2
  ) AS "percentageNew"
FROM period_customers

Mục đích nghiệp vụ & phân biệt CANCELLED vs RETURNED

Đơn CANCELLED là đơn bị huỷ trước khi giao — thường do hết hàng, buyer đổi ý, hoặc lỗi thanh toán. Đơn RETURNED là đơn đã giao xong nhưng buyer yêu cầu trả lại — thường do sản phẩm không như mô tả hoặc lỗi chất lượng.

Hai loại này có nguyên nhân và tác động khác nhau với seller: cancelled ảnh hưởng đến tỉ lệ hoàn thành đơn (completion rate) còn returned ảnh hưởng đến uy tín sản phẩm.

Quy tắc phân loại
CANCELLED → cancelTime IS NOT NULL
RETURNED → cancelTime IS NULL AND cancelReason IS NOT NULL
ALL → (cancelTime IS NOT NULL)
          OR (cancelTime IS NULL AND cancelReason IS NOT NULL)

const buildBaseQb = (applyTypeFilter = true) => {
  const qb = creatorContentRepository.createQueryBuilder('cc')
    .where('cc.creatorUsername IS NOT NULL')
    .andWhere('cc.contentId IS NOT NULL')
    .andWhere('cc.publishedAt BETWEEN :startTime AND :endTime')
    .andWhere('cc.createdBy = :userId');

  if (applyTypeFilter) {
    if (type === 'hasGmv') qb.andWhere('cc.totalPaidAmount > 0');
    if (type === 'noGmv')  qb.andWhere('cc.totalPaidAmount = 0');
  }
  return qb;
};

TypeORMCASE WHEN expression tái sử dụng trong 3 queries
const priceExpression = `
  CASE
    WHEN CAST(li.gift_retail_price AS DECIMAL) > 0
      THEN CAST(li.gift_retail_price AS DECIMAL)
    WHEN CAST(li.original_price AS DECIMAL) > 0
      THEN CAST(li.original_price AS DECIMAL)
    ELSE COALESCE(CAST(li.sale_price AS DECIMAL), 0)
  END`;

// Summary: tổng chi phí + tổng sample đã giao
const summaryQb = baseQb().select([
  `SUM(${priceExpression}) AS "totalCost"`,
  `COUNT(DISTINCT o.id) AS "totalSamples"`,
]);

// List: group by productId, tính cost và avg per sample
const listQb = baseQb()
  .select([
    'li.productId AS "productId"',
    `SUM(${priceExpression}) AS "totalCost"`,
    `AVG(${priceExpression}) AS "costPerSample"`,
    'COUNT(li.id) AS "quantity"',
  ])
  .groupBy('li.productId')
  .orderBy(sortExpression, order)
  .skip(skip).take(take);
